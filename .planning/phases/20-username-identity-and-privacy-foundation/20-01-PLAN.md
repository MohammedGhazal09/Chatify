# Plan 20-01: Backend Username Model, Validation, Indexing, And Migration-Safe Contract

## Objective

Create the backend username foundation: normalized unique usernames on `Users`, shared backend validation, signup enforcement, first-time setup API, and backend tests proving migration safety, uniqueness, auth, and CSRF behavior.

## Tasks

1. Read `20-SPEC.md`, `20-CONTEXT.md`, `20-UI-SPEC.md`, `userModel.mjs`, `authController.mjs`, `userController.mjs`, `userRouter.mjs`, and the existing auth/user/security tests.
2. Add `Backend/Chatify/Utils/usernameValidation.mjs` with normalization, reserved-name handling, validation result codes/messages, and a safe helper for controllers/tests.
3. Add `username` to `userModel.mjs` with trim/lowercase normalization, schema validation, and a unique sparse or partial index that allows existing users without username.
4. Update local signup in `authController.mjs` to require username, normalize it, reject invalid/reserved names, detect duplicate usernames as a conflict, and preserve existing email/password/session behavior.
5. Add a first-time username setup controller in `userController.mjs` that requires auth, rejects invalid/duplicate handles, refuses changes when a username already exists, and returns the account-safe current user.
6. Mount the setup endpoint in `userRouter.mjs` as a protected CSRF-enforced unsafe request, recommended path `PATCH /api/user/username`.
7. Add backend tests for username validation, model normalization, duplicate normalized usernames, signup required/invalid/duplicate/success behavior, auth required, CSRF required, first-time setup success, and second update rejection.

## Verification

- `cd Backend/Chatify; npm test -- --run test/auth/auth.lifecycle.test.mjs test/user/user.username.test.mjs test/security/csrf.test.mjs`
- Focused backend assertion proves an existing user document without `username` can still be loaded before setup.

## Success Criteria

The backend has a migration-safe username field and a protected first-time setup API, and every backend caller uses the same validation/normalization rules.

## Must Haves

- D-20-01: `Users.username` is normalized, unique, indexed, and migration-safe.
- D-20-02: backend validation uses one helper with the approved grammar and reserved-name list.
- D-20-03: normalization happens before uniqueness checks and case-only duplicates collide.
- D-20-04: username setup lives under the existing user route surface as `PATCH /api/user/username`.
- D-20-05: the setup endpoint accepts `{ username }`, requires auth and CSRF, succeeds only when username is missing, and returns the account-safe user.
- D-20-06: error/status behavior follows the locked `400`/`401`/`403`/`409` contract.
- D-20-07: do not introduce username login.
- D-20-08: local signup requires username while preserving existing email/password/session behavior.
- D-20-09: do not auto-generate usernames for OAuth or existing users.
- D-20-11: setup is first-time only; username changes remain out of scope.
- D-20-14: do not log raw username setup payloads, emails, tokens, OAuth payloads, or reset codes.
- D-20-18: backend tests cover normalization, validation, collisions, signup, first-time setup, CSRF, duplicate conflicts, and no second update.
- V2-USER-01, V2-USER-03, AUTH-01, AUTH-02, SEC-01, and SEC-02 behavior must not regress.
