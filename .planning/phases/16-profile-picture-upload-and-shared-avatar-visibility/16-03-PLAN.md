# Plan 16-03: Avatar Rendering Surfaces And Fixture Guardrails

<objective>
Render uploaded profile pictures across all required chat identity surfaces through a reusable fallback-safe avatar component, while preserving fixture guardrails against static demo identity content and storage internals.
</objective>

## Context

This plan depends on the backend and Settings mutation contract. It does not add new profile-picture behavior; it makes existing identity surfaces consume the safe `profilePic` data and fall back to `AbstractIdentityTile` when absent or when image loading fails.

## Context Decisions Covered

D-28 D-29 D-30 D-31 D-35 D-36 D-38 D-39

## Artifacts this plan produces

- Reusable `UserAvatar` component.
- Updated current-account sidebar, chat row, header, detail rail/drawer, and contact/search rendering.
- Component tests for image present, missing image, and failed image load.
- Narrow fixture leak guard updates.

<must_haves>
- Preserve `AbstractIdentityTile` as the fallback.
- Keep dimensions stable so image load/failure does not shift layout.
- Avoid editing `Frontend/Chatify/src/pages/chat/chat.tsx` unless no child component integration point exists.
- Do not introduce static demo avatars or production fixture identity images.
</must_haves>

<threat_model>
Assets: user identity imagery, chat identity UI, fixture guardrails, and private storage implementation details.
Threats: static demo avatars leaking into production runtime, broken image loads leaving blank identity surfaces, storage internals appearing in UI/tests, and layout shift across mobile or desktop.
Mitigations: a single fallback-safe `UserAvatar`, failed-load state tests, stable dimensions, narrow fixture guard allowlist, and storage-internal scans.
Residual risk: browser cache can temporarily retain older images until the safe version token changes; backend URL versioning should address this.
</threat_model>
