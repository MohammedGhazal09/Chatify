# Phase 10.1 Pattern Map

**Created:** 2026-06-13
**Mode:** Inline pattern map, no subagents per project instruction.

## Purpose

Map Phase 10.1 implementation targets to existing analogs so execution stays brownfield-safe and does not invent a parallel chat architecture.

## Backend Patterns

| Target | Existing Pattern | Keep / Change |
|--------|------------------|---------------|
| Message create validation | `Backend/Chatify/Controller/messageController.mjs` validates auth, chat access, attachments, text, and `clientMessageId` before `Message.create()` | Keep controller path; tighten missing `clientMessageId` handling |
| Idempotent create | `Message.findOne(idempotencyFilter)`, unique partial index, duplicate-key recovery | Keep; add concurrent test and no second emit assertion |
| Attachment identity | `validateIncomingAttachments()` and stored `attachmentFingerprint` | Keep; verify same/different attachment retry behavior |
| Canonical socket broadcast | `io.in(chat._id.toString()).emit('message:new', serializedMessage)` after first create | Keep; never broadcast for idempotent duplicate responses |
| Deprecated socket create | `socket.on('message:send', ...)` returns `deprecated_socket_message_send` | Keep; test it never persists |
| Delivery update | `message:delivered` resolves persisted message and membership before status update | Keep; ensure frontend calls it only after receive/cache |
| Join fallback | `chat:join` calls `markMessagesAsDelivered()` | Narrow via tests and semantics; active join may be fallback, passive auto-join must not mark delivered |
| User-specific unread | `emitToUserSockets(userId, 'unread:update', { count })` | Keep absolute count semantics |

## Frontend Patterns

| Target | Existing Pattern | Keep / Change |
|--------|------------------|---------------|
| API payload | `messageApi.createMessage()` sends JSON for text-only and FormData for attachments | Keep; assert all sends include `clientMessageId` |
| Optimistic creation | `createOptimisticMessage()` uses `optimistic-${clientMessageId}` and `status: 'sent'` | Keep |
| Cache merge | `upsertMessageInCache()` and `mergeCanonicalMessage()` merge message sources | Keep central; strengthen identity alias coverage |
| Send mutation | `useSendMessage()` owns optimistic insert, HTTP create, failure marking, success upsert, invalidation | Keep; make tests cover duplicate/retry paths |
| Page send handler | `chat.tsx` generates `clientMessageId` and calls `sendMessage.mutate()` | Add same-draft in-flight guard here or a small focused hook |
| Socket receive | `useChatSocket.ts` handles `message:new` and updates Query cache | Keep durable writes in hook |
| Page socket callback | `chat.tsx` callback currently calls `upsertMessage()` and `emitMessageDelivered()` | Convert callback to UI effects or move delivery ack into hook after cache upsert |
| Reconnect | `socket:ready`, `connect`, and `reconnect` invalidate/refetch chats, active messages, unread, presence | Keep; add no-duplicate tests |

## Test Patterns

| Surface | Existing Test Pattern | Phase 10.1 Extension |
|---------|-----------------------|----------------------|
| HTTP idempotency | `Backend/Chatify/test/message/message.idempotency.test.mjs` | Missing `clientMessageId`, concurrent duplicate creates, no second persisted doc |
| Attachment idempotency | `Backend/Chatify/test/message/message.attachments.test.mjs` | Same-id same-attachment duplicate and changed-attachment conflict |
| Socket state | `Backend/Chatify/test/socket/socket.message-state.test.mjs` | Recipient receives without refresh through real socket; offline recipient does not promote delivered |
| Socket auth | `Backend/Chatify/test/socket/socket.authorization.test.mjs` | Non-member receive/receipt rejection remains covered |
| API payloads | `Frontend/Chatify/src/api/messageApi.test.ts` | Text and multipart sends always carry `clientMessageId` |
| Cache convergence | `Frontend/Chatify/src/hooks/messageCache.test.ts` | Full optimistic -> HTTP -> socket -> refetch -> retry convergence |
| Query mutation | `Frontend/Chatify/src/hooks/useChatQueries.test.tsx` | Send variables, failure, retry, invalidation, one optimistic row |
| Socket hook | `Frontend/Chatify/src/hooks/useChatSocket.test.tsx` | Incoming cache upsert followed by delivery ack for other-user messages, no ack for own messages |
| Composer/page behavior | `MessageComposer.test.tsx` plus focused chat send test | Rapid duplicate Enter/click produces one mutation |
| Production smoke | `Frontend/Chatify/e2e/chat-production-reality.spec.ts` and `productionSmoke.ts` | Dedicated Phase 10.1 delivery reliability spec and artifact |

## Execution Landmines

- `Frontend/Chatify/src/pages/chat/chat.tsx` has recurring local work warnings. Read the file and current diff before editing; keep edits tightly scoped.
- Do not create a socket persistence path. HTTP create is the only write path.
- Do not rely on mocked Playwright routes for final delivery acceptance.
- Do not log message text or private payloads when adding diagnostics.
- Do not claim production success without smoke credentials/deploy identifiers or explicit blocked evidence.
