# Plan 15-03: Frontend WebRTC Controller And Call UI Repair

<objective>
Fix the frontend call controller and call UI so audio and video calls can connect reliably, early ICE is not lost, video failures remain explicit, and users get clear retry/end states without unrelated chat redesign.
</objective>

## Context

This plan depends on 15-01 failing tests and findings. It may redesign the call overlay and call controls as needed, but it must not refactor unrelated messenger layout or overwrite protected `chat.tsx` work.

## Artifacts this phase produces

- Frontend fixes mapped in `15-FAILURE-REPORT.md`.
- Updated call controller and WebRTC unit tests.
- Updated call overlay tests and, if UI changes are visual, Playwright screenshots/traces in acceptance artifacts.

<must_haves>
- No silent video-to-audio fallback.
- ICE candidates are buffered and flushed safely.
- Audio call and video call both have verified happy paths.
- Camera/setup failures show explicit error and retry states.
- Call redesign stays scoped to call controls and call overlay.
</must_haves>

<threat_model>
Assets: user media permission state, local/remote media streams, call session id, socket action acks, and user trust in call mode.
Threats: starting an audio call after a video request fails, dropping ICE candidates causing false failures, retaining camera/mic tracks after failure, stale call signals mutating a new call, and UI implying secure connection before media is connected.
Mitigations: explicit requested-mode tracking, no video fallback, call-id keyed pending signal buffers, media cleanup on all terminal paths, ack-aware errors, and accessible UI states.
Residual risk: fake media cannot prove all real device/browser combinations; production smoke and manual device checks remain useful.
</threat_model>

<task id="15-03-T1" title="Remove silent video-to-audio fallback">
<read_first>
- `Frontend/Chatify/src/utils/webrtcCallSession.ts`
- `Frontend/Chatify/src/hooks/useCallController.ts`
- `Frontend/Chatify/src/hooks/useCallController.test.tsx`
- `.planning/phases/15-investigate-and-fix-audio-and-video-call-reliability/15-FAILURE-REPORT.md`
</read_first>
<action>
Change media acquisition so `video` requests either return video media or throw a video-specific error. Do not retry audio automatically. Update controller handling so outgoing video does not emit `call:start` if video media fails, and incoming video does not emit `call:accept` if video media fails.
</action>
<acceptance_criteria>
- Existing fallback test is replaced by explicit video failure tests.
- `emitCallStart` is not called with `mode: 'audio'` after video failure.
- `emitCallAccept` is not called after incoming video media failure.
- Media tracks are stopped on failure and retry.
- UI error copy gives a clear camera/microphone action.
</acceptance_criteria>
</task>

<task id="15-03-T2" title="Implement call-id keyed ICE candidate buffering">
<read_first>
- `Frontend/Chatify/src/hooks/useCallController.ts`
- `Frontend/Chatify/src/utils/webrtcCallSession.ts`
- `Frontend/Chatify/src/hooks/useCallController.test.tsx`
- MDN `RTCPeerConnection.addIceCandidate` behavior summarized in `15-RESEARCH.md`
</read_first>
<action>
Add pending ICE buffering for candidates received before the peer session exists or before the peer can accept candidates. Key pending candidates by active call id and ignore stale call ids. Flush after peer creation and remote description setup for caller/callee paths. Clear pending candidates on reset, terminal sync, chat change, logout, and before unload.
</action>
<acceptance_criteria>
- Early ICE candidate tests pass for caller and callee.
- Stale call id candidates are ignored.
- Pending candidates do not leak across calls.
- `addIceCandidate` errors move call to explicit failed state and cleanup media.
- No raw candidate strings are logged.
</acceptance_criteria>
</task>

<task id="15-03-T3" title="Make offer/answer sequencing deterministic">

<action>

</acceptance_criteria>
