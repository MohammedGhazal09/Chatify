# Plan 16-01: Backend Profile Image Contract, Storage, And Security

<objective>
Implement the server-side profile image contract: safe user metadata, dedicated storage and validation, authenticated upload/replace/remove/fetch routes, CSRF/rate-limit protection, safe response payloads, and focused backend tests.
</objective>

## Context

Phase 16 depends on a stable backend contract before frontend work can safely proceed. The existing attachment implementation is useful precedent, but profile pictures are identity data with a smaller type/size boundary and a different lifecycle. This plan must not reuse the message attachment bucket or expose storage internals.

## Context Decisions Covered

## Artifacts this plan produces

- Dedicated profile-image validation utility.
- Dedicated profile-image storage service or bucket wrapper.
- User schema fields for provider image and uploaded override metadata.
- Protected upload, remove, and fetch routes.
- Safe `profilePic` response shaping across identity payloads.
- Backend tests proving validation, lifecycle, privacy, auth, CSRF, and rate-limit behavior.

<must_haves>
- Keep raw GridFS ids, bucket names, hashes, object keys, and private paths out of API responses.
- Do not trust local signup `profilePic`.
- Do not weaken existing message attachment privacy or tests.
- Do not log profile image bytes, filenames where unnecessary, storage ids, hashes, tokens, cookies, emails, or raw multipart metadata.
- Do not edit frontend chat route code in this plan.
</must_haves>

<threat_model>
Assets: profile image bytes, user identity metadata, auth cookies, CSRF tokens, provider image URLs, user discovery responses, and storage identifiers.
Threats: unauthenticated image streaming, CSRF upload/remove abuse, storage-id leakage, arbitrary external image injection through signup, orphaned images after failed replacement, and privacy leaks in logs or errors.
Mitigations: auth middleware on all profile-image routes, route-level CSRF for unsafe methods, dedicated limiter, server-side signature validation, safe app URLs, metadata cleanup on failure, and response-shape tests.
Residual risk: authenticated users who can already discover another user's identity can fetch the emitted profile image route; stricter relationship-only fetch can be considered later if discovery payloads are narrowed.
</threat_model>

- `Backend/Chatify/Models/userModel.mjs`
- `Backend/Chatify/Controller/messageController.mjs`
</read_first>
<action>
Implement replacement so new image bytes are stored, user metadata is persisted, and the previous uploaded image is deleted or deactivated best-effort afterward. Implement failure cleanup so newly uploaded bytes are removed if user metadata persistence fails. Implement removal so uploaded metadata is cleared and the previous uploaded image is no longer active.
</action>
<acceptance_criteria>
- Replacing image A with B makes payloads reference B and prevents A from being served as the current profile picture.
- Failed metadata update cleans up newly uploaded image data.
- Remove clears uploaded override and returns the effective provider image or no image.
- Best-effort old image cleanup failures are logged generically without sensitive metadata and do not break the successful new image response.
</acceptance_criteria>
</task>
