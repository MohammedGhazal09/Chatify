<threat_model>
Trust boundaries:
- Target email crosses from browser input into authenticated chat creation.
- Chat/message ids cross from route params into private backend resources.
- Search query text crosses into MongoDB matching.
- Search result rows cross from server state into browser UI without becoming durable history state.

STRIDE register:
- Spoofing: Client could target a chat id it does not belong to. Mitigation: search controller must load chat by id and verify requester membership.
- Tampering: Search query could inject regex behavior. Mitigation: escape regex metacharacters before constructing the query.
- Repudiation: Duplicate direct chats obscure which conversation is canonical. Mitigation: database-enforced `directKey` uniqueness with duplicate-key re-query.
- Information disclosure: New chat failures could reveal whether an account exists. Mitigation: generic failure message for missing user and self-target paths.
- Information disclosure: Search could return deleted or unauthorized messages. Mitigation: use requester visibility filters and exclude tombstones.
- Denial of service: Search could be spammed or unbounded. Mitigation: 2-character minimum, about 300 ms debounce, max 25 results, selected-chat scope only.
- Elevation of privilege: Frontend could show presence/profile data from a failed contact lookup. Mitigation: New chat shows only submitted email/error state until a valid chat response returns.
</threat_model>
