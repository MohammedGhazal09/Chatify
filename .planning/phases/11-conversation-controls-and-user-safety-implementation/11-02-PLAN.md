# Plan 11-02: Frontend Controls, Search, Detail Data, and Accessibility

## Objective

Replace static chat controls with real, stateful frontend behavior while preserving the approved Chatify visual design across desktop, mobile, light theme, and dark theme.

## Implementation Steps

1. Update frontend types and API methods.
   - Add `ConversationControls` and detail payload types.
   - Add typed `blockChatPeer(chatId)` and `unblockChatPeer(chatId)` methods.
   - Add or refine message/search/shared-asset API methods only if the existing endpoints cannot provide bounded data.

2. Wire TanStack Query hooks.
   - Add mutations for block and unblock.
   - Invalidate only selected chat, chat list, messages, shared detail, and search queries affected by the action.
   - Keep optimistic UI conservative: disable send immediately, but reconcile final state from server response.

3. Implement a real conversation More menu.
   - Create or update `ConversationMoreMenu`.
   - Open/close from header and details panel.
   - Close on Escape, outside click, chat change, route change, and successful action.
   - Support keyboard navigation and focus restoration.
   - Route menu items to real behavior or honest disabled states.

4. Make detail panel closable and data-driven.
   - Add pointer and keyboard close controls for the right panel.
   - Keep mobile and desktop state predictable when switching chats.
   - Render pinned messages from persisted pinned state.
   - Render shared files and shared media from real message attachments.
   - Hide or empty-state sections when data is absent.
   - Add bounded load-more for large pinned/shared sections.

5. Implement unified message search behavior.
   - Make header and detail-panel search open the same search workflow.
   - Search current loaded messages immediately.
   - For unloaded matches, use bounded `loadMoreMessages` before introducing any endpoint.
   - Highlight and scroll to the selected result without breaking virtual/scroll position.
   - Clear highlights when the query is cleared or the chat changes.

6. Respect conversation controls in the composer and actions.
   - Disable text send, attachment send, voice start, emoji send, typing emits, call attempts, reaction, edit, and delete-for-everyone when controls disallow active interaction.
   - Show a clear blocked-state reason from `messagingDisabledReason`.
   - Preserve history reading, selected-message actions that are allowed, and delete-for-self where implemented.

7. Make avatar/profile affordances honest.
   - If current profile update API exists, wire avatar/profile change through it.
   - If no safe capability exists, remove edit styling from the conversation peer avatar and show only view-profile behavior or a disabled menu item.

8. Add frontend unit/integration tests.
   - More menu open/close, keyboard behavior, and action dispatch.
   - Search workflow from both entry points.
   - Blocked composer disabled states.
   - Detail panel closes and renders empty/shared sections from props/query data.
   - Query invalidation targets for block/unblock.
   - Socket hook ignores suppressed/disallowed events after controls update.

## Acceptance Criteria

- Every visible conversation control either performs a real action or presents an honest disabled/unavailable state.
- The right details panel is closable by pointer and keyboard.
- Search messages works from all visible entry points and can jump to loaded or bounded-unloaded results.
- Block/unblock changes composer, menu, and realtime behavior without refreshing the page.
- Pinned messages, shared files, shared media, and security rows no longer render canned production fixtures.
- Mobile and desktop layouts preserve the approved visual direction in light and dark themes.

## Out of Scope

- Implementing a full voice/video calling backend if no call service exists.
- Group chat moderation.
- Broad visual redesign outside the existing approved Chatify messenger direction.
