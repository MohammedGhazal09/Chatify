# Plan 15-02: Backend Signaling, Session Authority, TURN, And Privacy Hardening

<objective>
Preserve and strengthen the backend call authority layer so call start, accept, sync, offer, answer, ICE, disconnect, multi-tab reachability, TURN readiness, and privacy behavior are reliable and test-proven.
</objective>

## Context

This plan depends on 15-01 findings. It should not redesign the frontend UI. Backend changes should be targeted and only made when tests or findings show a gap.

## Artifacts this phase produces

- Backend fix entries in `15-FAILURE-REPORT.md`.
- Backend socket/session tests proving call signaling and security behavior.
- TURN readiness evidence for `15-CALL-ACCEPTANCE.md`.

<must_haves>
- Server remains authoritative for call session state.
- Offer emission remains after callee accept/server connected sync.
- Socket identity remains cookie-authenticated and cannot be replaced by client events.
- Signals forward only to authorized peer sockets.
- Production call readiness is blocked unless TURN is configured.
</must_haves>

<threat_model>
Assets: call sessions, socket identity, chat membership, active call state, user reachability, TURN credentials, and call diagnostics.
Threats: unauthorized signaling into another chat, client-forged identity, rate-limit bypass, stale call resurrection, multi-tab misrouting, TURN credential leakage, and production readiness false positives.
Mitigations: keep socket middleware auth, load server call sessions for every action, validate signal payload bounds, use user socket maps for delivery, retain call event rate limits, and redact logs/artifacts.
Residual risk: server tests cannot prove browser media connectivity; frontend and Playwright plans cover that.
</threat_model>

<action>
Verify and, if needed, repair backend behavior so `call:offer`, `call:answer`, and `call:ice-candidate` are accepted only for an active server call where the actor is a participant and signaling is allowed. </action>
<acceptance_criteria>
- Unauthorized or stale signals receive stable failure ack codes.
- Offer before accept is rejected unless the server call is in the correct connected/signaling state.
- Authorized signals forward only to the peer user's sockets.
- Tests document the accepted-call-before-offer contract.
</acceptance_criteria>
</task>
