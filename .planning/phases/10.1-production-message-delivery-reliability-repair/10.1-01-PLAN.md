<decision_traceability>
- D-01: Keep repair targeted to backend message controller, socket layer, and existing tests.
- D-02: Preserve HTTP create plus Socket.IO broadcast/status architecture.
- D-04: Keep socket `message:send` deprecated and non-persistent.
- D-05: Reject browser-originated message creation without `clientMessageId`.
- D-06: Emit `message:new` and unread updates only for first successful create.
- D-07: Use the current unique partial index plus duplicate-key recovery unless tests prove it insufficient.
- D-08: Same-payload duplicate retries return canonical message; changed payload returns conflict.
- D-09: Attachment sends participate in the same idempotency contract.
- D-16: Recipient delivery ack happens after receive/cache; backend stays idempotent.
- D-17: Passive auto-join must not by itself imply delivered.
- D-18: HTTP success and optimistic render only justify `sent`.
- D-19: Multi-socket sessions must not duplicate receipt transitions.
- D-21: Non-member sockets must not receive messages or update receipts.
- D-22: Backend unread payloads are absolute server truth.
- D-25: Sender unread must not drift from send actions.
- D-26: Add backend idempotency/socket tests.
- D-32: Add focused redacted delivery lifecycle diagnostics.
</decision_traceability>

## Artifacts This Phase Produces

- A tightened `newMessage` validation branch that fails missing `clientMessageId`.
- Backend tests for missing id, concurrent duplicate create, attachment duplicate/conflict, socket no-create, offline/sent delivery truth, and receipt authorization.
- A small delivery lifecycle diagnostic helper or local logging wrapper if implementation needs shared redaction.

<threat_model>
- T-10.1-01: Duplicate HTTP requests race and create multiple persisted messages.
- T-10.1-02: Missing or changed `clientMessageId` bypasses frontend merge identity.
- T-10.1-03: Sender or non-member sockets falsely promote delivery/read state.
- T-10.1-04: Diagnostics leak private message content or credentials.

Mitigations: required client identity, unique index and duplicate-key recovery, same-payload conflict checks, membership-derived receipt updates, idempotent status transitions, and redacted metadata-only logs.
</threat_model>
