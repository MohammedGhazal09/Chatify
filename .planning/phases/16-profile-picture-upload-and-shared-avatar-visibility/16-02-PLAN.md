# Plan 16-02: Settings Profile Picture Workflow And Cache Propagation

<objective>
Add the authenticated Settings profile picture workflow, frontend API methods, mutation hook, immediate current-user update, and query invalidation needed for normal app propagation.
</objective>

## Context

This plan starts after the backend contract is available. It should consume the backend routes through `userApi` and a focused hook rather than calling Axios directly from `SettingsModal`. The UI scope is upload, preview, replace, remove, cancel/reset, loading, success, and errors only. Image editing tools are out of scope.

## Context Decisions Covered

D-15 D-24 D-25 D-26 D-27 D-32 D-33 D-34 D-36 D-39

## Artifacts this plan produces

- Frontend profile image API methods.
- Focused profile image mutation hook.
- Settings modal profile picture controls.
- Immediate auth store update and targeted query invalidation.
- Settings workflow tests.

<must_haves>
- Backend remains authoritative for validation and errors.
- Settings controls must be labelled and keyboard-operable.
- Do not add crop, zoom, rotate, filters, or image editor UI.
- Do not add a dedicated Socket.IO profile-image event.
- Avoid direct edits to `Frontend/Chatify/src/pages/chat/chat.tsx`.
</must_haves>

<threat_model>
Assets: selected local image file, preview object URL, auth user state, CSRF token, profile image response, and cached chat/contact data.
Threats: bypassing backend validation, leaking local object URLs after modal close, stale identity caches, inaccessible file controls, and direct Axios calls that skip shared CSRF handling.
Mitigations: client validation as advisory only, revoke object URLs, use shared Axios via `userApi`, centralize mutation side effects, and cover keyboard/label/error states in tests.
Residual risk: other users see changes only after normal refetch/refresh, which is accepted for Phase 16.
</threat_model>
