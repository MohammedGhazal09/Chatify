## Message Identity And Backend Idempotency

| Option | Description | Selected |
|--------|-------------|----------|
| Require client identity | Reject normal message creation without `clientMessageId`. | Yes |
| Tolerate missing identity | Allow server-created messages without a client merge key. | No |
| Emit only first create | Duplicate idempotent responses return existing message without socket re-emit. | Yes |
| Re-emit duplicates | Duplicate idempotent responses emit `message:new` again. | No |
| Unique index first | Use current unique index and duplicate-key recovery as concurrency boundary. | Yes |

**User's choice:** Approved recommendation.
**Notes:** Backend idempotency remains anchored to `(chatId, sender, clientMessageId)` plus normalized text and attachment fingerprint. Transactions are reserved for a proven orphan/consistency blocker.

## Frontend Send And Merge Contract

| Option | Description | Selected |
|--------|-------------|----------|
| Same-draft guard | Block only duplicate submissions of the same in-flight draft. | Yes |
| Global composer freeze | Disable all sending while any message is pending. | No |
| Central cache merge | Route durable message writes through shared cache/query/socket helpers. | Yes |
| Page-level competing writes | Allow `chat.tsx` callbacks and socket hook to both own durable cache writes. | No |
| Client id first | Merge by `clientMessageId` first, durable `_id` second. | Yes |

**User's choice:** Approved recommendation.
**Notes:** The duplicate-submit guard should prevent rapid Enter/click races without making the composer feel frozen for intentional changed drafts.

## Realtime Delivery And Receipt Semantics

| Option | Description | Selected |
|--------|-------------|----------|
| Recipient-confirmed delivery | Promote delivered after recipient receive/cache/ack or selected-chat join fallback. | Yes |
| HTTP-success delivery | Promote delivered after sender create request succeeds. | No |
| Auto-join receives only | Auto-joined rooms receive events but do not mark delivery by themselves. | Yes |
| Auto-join marks delivered | Mark all sent messages delivered simply because a socket connected and joined rooms. | No |
| No offline queue | Keep failed HTTP sends visible and manually retryable. | Yes |
| Hidden offline queue | Queue sends client-side while offline. | No |

**User's choice:** Approved recommendation.
**Notes:** This directly targets the false delivered state shown in production. Delivery/read UI must follow server patches only.
