# Plan 18-01: Structured Diagnostics And Redaction Layer

## Objective

Replace ad hoc operational logging on backend runtime boundaries with structured redacted events that a maintainer can grep without exposing secrets or private chat content.

## Tasks

1. Read `18-CONTEXT.md`, `requestLogger.mjs`, `DBConfig.mjs`, `queueMiddleware.mjs`, `requestQueue.mjs`, `socket.mjs`, and the current message/chat controller socket emit error sites.
2. Add `Backend/Chatify/Utils/observabilityLogger.mjs` with log levels, stable `event` field, ISO timestamp, safe error serialization, metadata redaction by key, and helpers for `info`, `warn`, and `error`.
3. Update request logging so every request receives `req.requestId`, `X-Request-Id`, structured start/finish/error events, safe method/path/status/duration fields, and no body/cookie/header dumps.
4. Replace queue, database, socket lifecycle, and selected emit-failure `console.*` calls with logger calls that include safe ids, event names, status codes, and error summaries only.
5. Add logger unit tests proving tokens, cookies, auth headers, passwords, reset codes, SDP, ICE candidates, email addresses, and nested secret-shaped fields are redacted.

## Verification

- `Backend/Chatify/Utils/observabilityLogger.mjs` exists and exports redaction and logger helpers.
- `requestLogger.mjs` sets `X-Request-Id` and logs structured objects.
- `npm test -- --run test/observability/observability-logger.test.mjs` passes from `Backend/Chatify`.
- A targeted grep of touched logging code finds no raw `console.log` calls added by this plan.

## Success Criteria

Backend diagnostic output is structured and redacted at the most important request, queue, database, socket, message, and call boundaries.

## Must Haves

- Logs must never include raw tokens, cookies, reset codes, OAuth payloads, SDP, ICE candidates, passwords, raw emails, or message text.
- Request and socket events must include only safe operational identifiers and event codes.
