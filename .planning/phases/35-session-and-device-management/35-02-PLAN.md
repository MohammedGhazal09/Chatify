# Phase 35 Plan 02 - Session-Bound HTTP And Socket Auth

## Objective

Bind new access tokens to active sessions and reject revoked sessions consistently across HTTP and Socket.IO authentication.

## Scope

- Session id access-token claim.
- Shared active-session validation helper.
- HTTP `protect` middleware session validation.
- Socket.IO handshake session validation.
- Refresh rotation metadata/session carry-forward.
- Backend/socket tests for revoked-session rejection.

## Acceptance Criteria

- [ ] New access tokens include a session id.
- [ ] Protected HTTP requests reject revoked or expired session ids.
- [ ] Socket handshakes reject revoked or expired session ids.
- [ ] Legacy access tokens without session id keep current expiry-based behavior.
- [ ] Tests cover refresh, revoke, HTTP rejection, and socket rejection.
