# Phase 19 Research

## Notification Findings

- The app has sound notification primitives in `Frontend/Chatify/src/utils/sounds.ts`, but the preference is global localStorage and not explicitly scoped to the authenticated user.
- `SettingsModal.tsx` has a sound toggle, which is the right entry point for notification controls. It should become a notification section with sound, browser notifications, permission state, and explanatory copy.
- `useChatSocket.ts` is the correct runtime boundary for realtime message alerts because it already receives `message:new`, updates caches, emits delivery receipts, and knows the selected chat id.
- The current socket handler returns early for non-selected chats before playing sound. Phase 19 should intentionally define background-message alerting instead of inheriting that behavior accidentally.
- Browser notification content should be generated by a dedicated privacy helper, not by sidebar snippets, latest message text, or message search labels.
- Conversation muting can start as client-local per-user preference. It should suppress alerting but still let unread state, message cache updates, and delivery/read receipts flow normally.
- True push delivery after the tab is closed requires service worker and push-subscription design; that belongs to v2, not this phase.

## Account And Session Findings

- `Frontend/Chatify/src/api/axios.ts` already dispatches `chatify:auth-expired` when refresh recovery fails.
- `Frontend/Chatify/src/pages/chat/chat.tsx` listens for auth expiry and hides private chat content before routing to login.
- `useLogout` clears the query client and auth state in the current tab, but multi-tab propagation needs an explicit browser-channel or storage-event contract.
- `useSelectedChatPersistence.ts` already prevents cross-user selected-chat leakage by namespacing localStorage keys with user id. Notification preferences should use the same principle.
- The existing session-expired view is recoverable, but account/logout/expired-session states should be consistent across Settings, sidebar, protected routes, and chat.

## State Surface Findings

- `ChatStateView.tsx` is the existing shared primitive for centered chat states.
- `ConversationPane.tsx` covers selected-chat empty state, session-expired state, offline/reconnecting banners, and blocked notices.
- `MessageList.tsx` covers loading, error, empty message, retry, and scroll-to-bottom behavior.
- `MessageSearchResults.tsx` covers below-minimum, loading, error, empty, loaded-result, and unloaded-result states.
- `MessageComposer.tsx` covers send disabled, failed send, upload, and voice-control states.
- `ConversationHeader.tsx` and `ConversationMoreMenu.tsx` already expose disabled call reasons. Phase 19 should make unavailable-call states visually and accessibly consistent rather than adding unsupported call behavior.
- Existing tests cover many isolated states, but the language, tone, live-region behavior, and mobile layout should be reviewed as one product system.

## Test Strategy Findings

- React Testing Library should cover hooks, Settings controls, socket alert routing, ChatStateView variants, sidebar mute indicators, and session broadcast behavior through accessible queries.
- `useChatSocket.test.tsx` already has a strong mocked Socket.IO harness and is the best unit test location for alert routing decisions.
- `SettingsModal.test.tsx` already mocks profile workflows and can cover notification controls if the modal remains the settings entry point.
- Playwright should cover behavior-first flows: notification settings states, muted/unmuted background alerts, auth-expired privacy, first-run/no-results/offline state copy, and mobile layout.
- Tests must avoid embedding realistic emails, tokens, private message text in browser notification assertions, screenshots, or generated evidence.

## Recommendation

Build a small notification domain first: typed preferences, privacy-safe copy, per-user persistence, permission helpers, and muted-chat helpers. Then wire those helpers into the existing Settings and socket surfaces with focused tests before touching broader chat UI polish.
