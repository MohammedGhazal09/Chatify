# Plan 11-03: Integrated Verification and Static Fixture Guard

## Objective

Prove Phase 11 behavior end to end and record evidence that the chat interface is functional, not a static picture.

## Implementation Steps

1. Add E2E coverage for conversation controls.
   - Open and close the right details panel on desktop and mobile.
   - Open More, verify keyboard behavior, trigger search from menu, and close menu.
   - Block a direct-chat peer and verify composer, attachment, voice, typing, and send controls become unavailable.
   - Unblock and verify active messaging controls recover.
   - Verify call/video controls are functional if capability exists or honestly disabled if not.

2. Add E2E coverage for realtime block behavior.
   - Use two authenticated browser contexts.
   - User A blocks User B.
   - User B cannot deliver a new message to User A through the UI.
   - User A does not receive typing, read/delivery, or other active signals from User B after block.
   - Existing history remains visible to both sides.

3. Add E2E coverage for search and detail data.
   - Search loaded messages from header and detail panel.
   - Search a result that requires bounded history loading.
   - Verify selected result highlight/scroll.
   - Verify pinned/shared/media sections are populated from real created messages and empty when no matching data exists.
   - Verify shared asset load-more behavior with enough seeded assets.

4. Add static-fixture guard tests.
   - Fail if production chat sections render canned pinned text or canned attachment/media names when the API returns empty data.
   - Keep explicit test fixture data inside test files or factories only.
   - Search production source for old sample labels before closing the phase.

5. Run required quality gates.
   - Backend tests for block/control helpers and socket suppression.
   - Frontend lint.
   - Frontend build.
   - Frontend unit/integration tests.
   - Playwright controls/search/block smoke in desktop and mobile sizes.
   - Manual or automated light/dark screenshot check when the UI has changed.

6. Write evidence.
   - Create `11-CONTROLS-EVIDENCE.md`.
   - Include commands, pass/fail results, screenshots if available, and known gaps.
   - Explicitly state whether Phase 10.1 live two-user delivery smoke is green or still blocking production readiness.

## Acceptance Criteria

- E2E tests prove the details panel, More menu, search, block/unblock, blocked composer, shared data, and empty states.
- Two-user realtime tests prove a blocked peer cannot create new active activity without a refresh.
- Static fixture guard prevents reintroducing canned pinned/shared/media/security content.
- Evidence artifact includes exact commands and results.
- The phase is not marked production ready unless Phase 10.1 live two-user smoke is also green.

## Verification

Run the final command set from the relevant package directories:

```powershell
# Backend/Chatify
npm test -- --runInBand

# Frontend/Chatify
npm run lint
npm run build
npm test -- --run
npx playwright test e2e/chat-conversation-controls.spec.ts
```

If local environment credentials prevent full live smoke, record the blocker explicitly in the evidence file and keep the phase open.

## Out of Scope

- Shipping production without the Phase 10.1 delivery smoke.
- Replacing the visual design system.
- Adding unrelated account, OAuth, or notification features.
