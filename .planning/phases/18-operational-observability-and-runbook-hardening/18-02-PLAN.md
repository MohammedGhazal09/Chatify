# Plan 18-02: Health And Readiness Endpoints

## Objective

Expose secret-safe health and readiness endpoints that report process, database, env, storage, queue, socket, CORS/cookie, and call/TURN readiness without making production blockers disappear.

## Tasks

1. Read `18-CONTEXT.md`, `app.mjs`, `DBConfig.mjs`, `socket.mjs`, `queueMiddleware.mjs`, `callIceConfig.mjs`, and backend env examples.
2. Add `Backend/Chatify/Utils/operationalReadiness.mjs` that builds liveness and readiness payloads with component statuses, missing env names, database ready state, queue status, storage readiness, socket initialized state, frontend origin posture, cookie posture, and call/TURN readiness.
3. Mount `GET /api/health` and `GET /api/ready` before protected routers. `/api/health` returns 200 when the process is alive. `/api/ready` returns 200 for ready, 503 for blocked/down, and includes only sanitized metadata.
4. Expose a safe socket initialization status helper from `socket.mjs` for readiness checks.
5. Update `.env.example` only for missing documented readiness-related variables, using placeholders and no secrets.
6. Add backend Supertest coverage for healthy, degraded, and missing-env readiness behavior.

## Verification

- `GET /api/health` returns `{ status: "ok" }` with a timestamp.
- `GET /api/ready` includes `database`, `environment`, `storage`, `socket`, `cors`, `cookies`, `queues`, and `calls` components.
- `GET /api/ready` never returns env values, tokens, cookies, TURN credentials, or database URLs.
- `npm test -- --run test/observability/health-readiness.test.mjs` passes from `Backend/Chatify`.

## Success Criteria

Operators can distinguish alive, degraded, and release-blocked states from one endpoint without leaking secrets.

## Must Haves

- Production readiness must fail closed when required production env values are missing.
- Missing TURN must be visible and production-blocking, not hidden behind local STUN fallback.
