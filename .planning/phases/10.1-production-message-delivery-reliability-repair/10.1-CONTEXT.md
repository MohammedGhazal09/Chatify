# Phase 10.1: Production Message Delivery Reliability Repair - Context

**Gathered:** 2026-06-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 10.1 repairs the live message delivery contract that production exposed as broken: one browser send must create one persisted message, one sender render, one realtime recipient render, and honest server-confirmed delivered/read state. The phase hardens existing HTTP message creation, `clientMessageId` idempotency, frontend cache merge, Socket.IO receive/receipt behavior, reconnect reconciliation, production smoke evidence, and redacted delivery diagnostics. It does not implement new messenger controls such as calls, video, block/unblock, voice messages, profile image editing, or side-rail feature work.

</domain>

**In scope (from SPEC.md):**
- Backend message creation idempotency for text and existing attachment sends.
- Frontend optimistic, HTTP, socket, retry, refetch, pagination, and reconnect message merge behavior.
- Composer duplicate-submit prevention for a single draft.
- Realtime `message:new` delivery to authenticated chat members.
- Server-truth `sent`, `delivered`, and `read` state requirements.
- Conversation list latest-message and unread-count consistency tied to message delivery.
- Security checks that prevent non-members from receiving messages or updating receipts.
- Local automated tests and two-account smoke coverage for the delivery contract.
- Deployed smoke evidence or an explicit blocked status if production credentials/deploy access are unavailable.
- Redacted delivery observability needed to diagnose this reliability class.

**Out of scope (from SPEC.md):**
- Call and video-call implementation - separate feature surface, not required to repair message delivery.
- Block/unblock user actions - separate conversation control phase.
- Profile image changing and user avatar management - separate account/profile feature.
- Side rail close/search/more UX fixes - real functional gaps, but not part of the urgent delivery reliability contract.
- New voice-message recording implementation - separate media/composer feature.
- New media gallery or shared-file UX features beyond refreshing existing side effects from canonical messages.
- Automatic destructive cleanup of historical duplicate production messages - data repair must be planned separately after future duplicates are prevented.
- Replacing React/Vite, Express, MongoDB/Mongoose, Socket.IO, TanStack Query, Zustand, or Tailwind - keep the existing stack.
- Re-enabling socket-based message persistence - one create path is required for this phase.

<decisions>
## Implementation Decisions

### Repair Strategy And Scope Control
- **D-01:** Use a targeted reliability repair, not a broad chat rewrite. Focus expected changes around `Backend/Chatify/Controller/messageController.mjs`, `Backend/Chatify/Config/socket.mjs`, `Frontend/Chatify/src/hooks/messageCache.ts`, `Frontend/Chatify/src/hooks/useChatQueries.ts`, `Frontend/Chatify/src/hooks/useChatSocket.ts`, and the send orchestration in `Frontend/Chatify/src/pages/chat/chat.tsx`.
- **D-02:** Preserve the existing architecture: HTTP message creation, Socket.IO broadcast/status events, TanStack Query durable cache, Zustand auth/presence stores, and componentized chat UI.
- **D-03:** Do not include calls, video, block/unblock, voice, profile image editing, side-rail feature work, or historical production data cleanup in Phase 10.1.
- **D-04:** Keep socket `message:send` deprecated and non-persistent. No socket event should create a message in this phase.

### Message Identity And Backend Idempotency
- **D-05:** Reject normal authenticated browser message creation without `clientMessageId` instead of allowing unmergeable server-created messages.
- **D-06:** Only the first successful create for a `(chatId, sender, clientMessageId, normalized text, attachment fingerprint)` emits `message:new` and unread updates. Idempotent duplicate responses return the existing message without a second socket emit.
- **D-07:** Use the existing unique partial index plus explicit duplicate-key recovery as the primary concurrent duplicate protection. Add transactions only if planning/execution tests expose orphaned attachment or consistency failures that the current pattern cannot close.
- **D-08:** Keep same-payload duplicate retries idempotent and return the canonical message. Same `clientMessageId` with different normalized text or attachment fingerprint remains a conflict.
- **D-09:** Existing attachment sends participate in the same idempotency contract. Retry attachment sends only while local `File` objects remain available.

### Frontend Send And Merge Contract
- **D-10:** Add a page-level or focused-hook in-flight duplicate-submit guard keyed by selected chat id, normalized text, and attachment file signatures.
- **D-11:** Prevent duplicate submission of the same in-flight draft while still allowing an intentionally changed draft after the first draft is registered.
- **D-12:** Centralize durable message cache writes through `messageCache.ts`, `useChatQueries.ts`, and `useChatSocket.ts`. Page callbacks should trigger UI side effects only, not perform competing durable cache writes.
- **D-13:** Preserve identity priority of `clientMessageId` first and durable `_id` second, then strengthen tests for optimistic, HTTP, socket echo, retry, refetch, pagination, missing/late id, and server-id aliasing cases.
- **D-14:** Sender conversation list can update optimistically for responsiveness, but it must reconcile from canonical `message:new` or refetch. Recipient conversation list updates only from server/socket/refetch truth.
- **D-15:** Failed sends remain visible as failed optimistic messages. Retrying unchanged content reuses the original `clientMessageId`; changed text or changed attachments are a new message identity.

### Realtime Delivery And Receipt Semantics
- **D-16:** Recipient clients should emit or acknowledge `message:delivered` only after receiving and caching an incoming `message:new`; selected-chat `chat:join` delivery sweep remains a reconnect fallback.
- **D-17:** Auto-joining all user chat rooms on socket connection should receive room events but should not, by itself, mark all messages delivered.
- **D-18:** HTTP create success and optimistic render only justify `sent`. The UI must show `delivered` or `read` only after server receipt/read patches.
- **D-19:** Multi-tab and multi-device same-account sessions are supported through the same canonical socket event and cache merge path. Multiple sockets must not create duplicate bubbles or duplicate receipt transitions.
- **D-20:** Offline sending does not add a hidden client-side send queue in this phase. Failed HTTP sends remain failed optimistic messages with manual retry.
- **D-21:** Non-member sockets must not receive `message:new`, must not mark messages delivered/read, and must remain covered by socket authorization tests.

### Conversation List, Unread, And Reconnect Reconciliation
- **D-22:** Treat backend `unread:update` payloads as absolute server truth, not client-side increments/decrements.
- **D-23:** On `socket:ready`, reconnect, and selected-chat join, invalidate/refetch chats, unread counts, and active chat messages, then merge into existing cache identities without duplicates.
- **D-24:** Reconnect reconciliation must repair missed messages and status patches without producing duplicate message rows or duplicate chat rows.
- **D-25:** Sender unread must not drift from send actions. Recipient unread changes only from server-backed events/refetches.

### Verification, Production Evidence, And Observability
- **D-26:** Add backend idempotency/socket tests, frontend cache/send tests, local two-account browser smoke with real local backend/API/socket, and opt-in production smoke.
- **D-27:** Do not rely on mocked Playwright routes for the Phase 10.1 delivery smoke. Mocked UI tests can support unit/UI confidence, but delivery acceptance must hit real HTTP and Socket.IO paths.
- **D-28:** Create a dedicated Phase 10.1 evidence artifact, `10.1-DELIVERY-RELIABILITY.md`. Use Phase 10 audit as baseline context, not as the final Phase 10.1 proof.
- **D-29:** Reuse Phase 10 production smoke config helpers where practical, but add a dedicated Phase 10.1 delivery reliability spec or command so general production audit and delivery acceptance stay separate.
- **D-30:** If production smoke credentials or deploy access are unavailable, mark deployed smoke blocked. Do not call the phase complete from local evidence alone.
- **D-31:** Verification must record frontend/backend URLs and deploy identifiers when available. If deploy identifiers are unavailable, record that as a residual evidence gap or blocker.
- **D-32:** Add focused redacted delivery lifecycle diagnostics rather than a broad logging framework. Logs may include request/event id, chat id, message id, `clientMessageId`, user role, lifecycle stage, and status transition metadata; logs must not include message text, attachment contents, cookies, tokens, emails, reset codes, OAuth payloads, or raw private storage ids.

### the agent's Discretion
- The planner may choose exact helper names, test file names, and local module boundaries if the locked reliability contract is preserved.
- The planner may decide whether the duplicate-submit guard lives directly in `chat.tsx` or in a focused hook, as long as unrelated local work is not overwritten and the route remains reviewable.
- The planner may decide whether delivery acknowledgement is implemented as a direct existing `message:delivered` emit after incoming cache upsert or a small helper wrapper around that event, as long as false delivery is prevented.
- The planner may choose exact production smoke spec names and artifact screenshot/trace names if the live/no-mock boundary is obvious.

</decisions>

## Existing Code Insights

### Reusable Assets
- `messageController.mjs`: Already implements HTTP create, idempotent lookup, duplicate-key recovery, attachment fingerprint conflict checks, unread emits, and `message:new`; extend rather than adding a second write path.
- `messageModel.mjs`: Already has a `clientMessageId` field and partial unique index; use it as the concurrency boundary.
- `socket.mjs`: Already authenticates sockets, joins rooms, emits `socket:ready`, keeps `message:send` deprecated, checks membership for delivery, and has delivery sweep logic; adjust semantics rather than moving socket logic into UI.
- `messageCache.ts`: Already owns canonical merge, optimistic replacement, failed retry replacement, receipt monotonicity, and fetched-message reconciliation; make all paths use it consistently.
- `useSendMessage`: Already performs optimistic insert, HTTP create, failure marking, and success upsert; add send identity enforcement and duplicate-submit guard integration.
- `useChatSocket`: Already handles `message:new`, `message:status-update`, `message:read`, `messages:read-batch`, `unread:update`, `socket:ready`, and reconnect invalidation; keep durable socket cache writes here.
- `chat-production-reality.spec.ts` and `productionSmoke.ts`: Existing production smoke patterns can be reused, but Phase 10.1 needs its own delivery reliability artifact and pass/fail gate.
- Existing backend/frontend tests cover many pieces but need expansion for the exact production bug: one send, one sender bubble, recipient receives without refresh, and status is not falsely delivered.

### Established Patterns
- Backend routes stay thin; controllers and helpers own validation, membership checks, persistence, and response shaping.
- Frontend pages/components consume typed hooks and APIs; durable server state is owned by TanStack Query hooks.
- Socket.IO room events are scoped by chat membership and must not trust client-supplied identity.
- Status transitions are monotonic and receipt patches should not downgrade read/delivered state.
- Failed sends remain visible and recoverable, but attachment retries require local `File` objects.
- Production smoke is opt-in and uses redacted env-provided smoke accounts.
- Production evidence cannot be substituted with mocked Playwright routes.

### Integration Points
- Tighten missing `clientMessageId` validation in the backend create path and frontend payload tests.
- Add duplicate-key and concurrent-create tests around existing idempotency helpers.
- Add duplicate-submit guard around `handleSendMessage` or a focused hook used by `chat.tsx`.
- Ensure sender socket echo and page callbacks do not create competing cache writes.
- Adjust delivery marking so auto room join does not falsely mark all messages delivered.
- Add recipient-side delivery emit/ack after incoming message cache upsert, with reconnect/join sweep as fallback.
- Add Phase 10.1 production delivery smoke and evidence artifact under `.planning/phases/10.1-production-message-delivery-reliability-repair/`.

<specifics>
## Specific Ideas

- The production bug to reproduce and close is concrete: one sender sends one message, sender must see one bubble, recipient must see the message without refresh, and sender must not see delivered until recipient-confirmed state.
- The prior Phase 10 production smoke can stay as the general production audit baseline; Phase 10.1 gets a dedicated delivery reliability proof artifact.
- Keep implementation small and defensible. The problem is contract drift across existing layers, not absence of an architecture.
- Treat production smoke credentials and deploy metadata as sensitive; use redacted account labels and never write passwords or real emails into artifacts.

</specifics>
