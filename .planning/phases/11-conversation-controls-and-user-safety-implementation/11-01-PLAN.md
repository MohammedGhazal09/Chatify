# Plan 11-01: Backend Conversation Controls and Block Enforcement

## Objective

Make conversation control state authoritative on the server and enforce directed user blocking across HTTP and Socket.IO paths before the frontend wires interactive controls.

## Implementation Steps

1. Add `UserBlock` persistence.
   - Create `Backend/Chatify/Models/userBlockModel.mjs`.
   - Fields: `blocker`, `blockedUser`, optional `sourceChatId`, timestamps.
   - Indexes: unique `{ blocker: 1, blockedUser: 1 }`, plus lookup indexes for both users.
   - Reject self-block at helper/controller level.

2. Add shared conversation-control helpers.
   - Create or extend backend helpers that can:
     - assert the requester belongs to a chat
     - confirm the chat is a direct chat before block/unblock
     - resolve the peer user id
     - fetch block state in both directions
     - serialize `conversationControls`
     - answer whether a requested action is allowed
   - Keep helpers independent of Express so controllers and sockets share them.

3. Implement chat-scoped block endpoints.
   - Add `POST /api/chat/:chatId/block`.
   - Add `DELETE /api/chat/:chatId/block`.
   - Return the updated chat/control payload expected by the frontend.
   - Reject group chats, non-members, self-block, malformed ids, and unauthenticated requests.
   - Make already-blocked and already-unblocked outcomes deterministic and safe.

4. Attach `conversationControls` to chat API responses.
   - Update chat list/detail response shaping so each direct chat has capability state.
   - Preserve existing chat fields and avoid broad response shape churn.
   - Keep group chat controls conservative until a future group-moderation phase.

5. Enforce block state in message HTTP paths.
   - Prevent new message persistence when either side blocks the other.
   - Return a clear 403-style response with a stable reason code/message.
   - Preserve history fetch, file download metadata, delete-for-self, and allowed read-only actions.

6. Enforce block state in Socket.IO paths.
   - Reject socket-originated `newMessage` before persistence and room broadcast.
   - Suppress active cross-user events after block: typing, presence/last-seen, delivery/read, reaction, pin/unpin, edit, delete-for-everyone, and future call attempts if present.
   - Allow room membership needed for history synchronization and self-only state.
   - Emit a targeted state update so both affected clients can refresh controls without receiving disallowed activity.

7. Add backend tests.
   - Model/helper tests for indexes, self-block rejection, direct-chat peer resolution, and both block directions.
   - Controller tests for block/unblock success, idempotency, non-member rejection, group rejection, and controls serialization.
   - Message tests for blocked HTTP sends and preserved history access.
   - Socket tests for blocked send rejection and suppressed active events.

## Acceptance Criteria

- A user can block and unblock another user from an existing direct chat through backend endpoints.
- Chat responses expose accurate `conversationControls` for the current viewer.
- Blocked users cannot create new messages through HTTP or Socket.IO.
- Active realtime events are suppressed across a blocked pair while history remains accessible.
- Backend tests cover both `blockedByMe` and `blockedMe` directions.
