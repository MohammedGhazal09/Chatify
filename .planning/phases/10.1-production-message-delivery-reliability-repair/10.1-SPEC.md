# Phase 10.1: Production Message Delivery Reliability Repair - Specification

**Created:** 2026-06-13
**Ambiguity score:** 0.07 (gate: <= 0.20)
**Requirements:** 13 locked

## Goal

Chatify message delivery changes from production-unreliable behavior to a server-truth contract where one user send creates exactly one persisted message, exactly one sender render, exactly one realtime recipient render, and delivery/read indicators reflect recipient-confirmed server state.

## Background

Phase 10 production review found critical reliability failures in the live chat experience: one send can render duplicate sender messages, the recipient may not receive the message until refreshing, and the sender UI can show delivered state before the recipient actually receives the message. This blocks later messenger feature work.

The current code already has several intended reliability pieces. `Backend/Chatify/Controller/messageController.mjs` creates messages through the HTTP `newMessage` controller, accepts `clientMessageId`, persists `status: 'sent'`, updates chat latest message, and emits `message:new` through Socket.IO. `Backend/Chatify/Models/messageModel.mjs` has a partial unique index for `chatId`, `sender`, and `clientMessageId`. `Backend/Chatify/Config/socket.mjs` authenticates sockets, joins chat rooms, emits `socket:ready`, handles `message:delivered`, and marks messages delivered when a recipient joins a chat. `Frontend/Chatify/src/hooks/useChatQueries.ts`, `Frontend/Chatify/src/hooks/messageCache.ts`, and `Frontend/Chatify/src/hooks/useChatSocket.ts` already contain optimistic send, cache merge, socket listener, receipt patch, retry, and reconnect-related behavior.

The gap is that these pieces do not yet prove a reliable end-to-end production contract. Phase 10.1 locks the message creation, merge, realtime receive, retry, reconnect, receipt, and verification requirements that must be true before work resumes on side-panel actions, calls, voice, media UX, profile-image editing, block actions, or other messenger controls.

## Requirements

1. **Single-send idempotency**: One browser send action creates one persisted message for the sender, even if the request is retried, duplicated, raced, or echoed through realtime.
   - Current: Backend idempotency exists by `clientMessageId`, but production shows one user send can render duplicate messages.
   - Target: Repeated or concurrent sends with the same sender, chat, `clientMessageId`, text, and attachment fingerprint converge to one canonical message id and one stored document.
   - Acceptance: A backend test sends the same payload at least twice, including a concurrent/race path, and verifies exactly one `Message` document exists, both responses resolve to the same message id, and no second `message:new` event creates a duplicate sender render.

2. **Required client message identity**: Every new browser-originated message send has a stable `clientMessageId` that survives optimistic render, HTTP response, socket echo, retry, refetch, and pagination.
   - Current: The frontend usually generates `clientMessageId`, but the reliability contract is not enforced as a phase-level requirement.
   - Target: Text sends and attachment sends always include `clientMessageId`; authenticated message creation without it is rejected for the normal chat send path.
   - Acceptance: API and mutation tests prove text-only and attachment-only sends include `clientMessageId`, and a backend test proves missing `clientMessageId` returns a validation failure instead of creating an unmergeable message.

3. **Single canonical create path**: HTTP message creation remains the only persistence path for new messages in this phase.
   - Current: The socket server has a deprecated `message:send` no-op, while HTTP creation persists and broadcasts messages.
   - Target: No socket event directly creates a message; socket events broadcast, reconcile, and update status only after HTTP persistence succeeds.
   - Acceptance: Socket tests prove `message:send` does not persist a message, while HTTP create persists once and emits the canonical `message:new` payload.

4. **Canonical frontend merge contract**: All message sources merge by `clientMessageId` first and durable message id second, without duplicate bubbles.
   - Current: `messageCache.ts` has merge helpers, but production still shows duplicate sender messages.
   - Target: Optimistic inserts, HTTP mutation success, sender socket echo, recipient socket receive, failed retry resolution, selected-chat refetch, chat pagination, and reconnect reconciliation all use the same identity semantics.
   - Acceptance: Frontend tests feed the same logical message through optimistic, HTTP, socket, retry, refetch, and pagination paths and verify one final message object with the canonical server id, preserved status fields, and no duplicate rendered row.

5. **Composer duplicate-submit guard**: One composer submission cannot launch multiple in-flight create mutations for the same draft before the first mutation is registered.
   - Current: The page-level send handler creates a new `clientMessageId` per call, so rapid Enter/click paths can create multiple sends if not guarded.
   - Target: Rapid double-click, Enter spam, or button-plus-keyboard races for the same draft produce one send attempt unless the user intentionally submits a new draft after the composer is reset.
   - Acceptance: A frontend interaction test triggers rapid duplicate submit events and verifies exactly one mutation call and one optimistic message for the draft.

6. **Retry contract**: Retrying a failed message reuses the original `clientMessageId` and never creates a second logical message for unchanged content.
   - Current: Failed-message retry reuses `clientMessageId`, but this needs to be locked and verified across text and attachment cases.
   - Target: Retry with the same text and same available local attachments resolves the failed optimistic message into the canonical persisted message; changed text or changed attachments are treated as a new send with a new identity.
   - Acceptance: Tests verify failed text retry and failed attachment retry converge to one message, while changed payload with reused `clientMessageId` receives a conflict and does not create a second message.

7. **Attachment idempotency**: Existing attachment/media sends participate in the same message identity contract.
   - Current: Attachment fingerprinting and validation exist, but production reliability scope must include current attachment sends.
   - Target: Attachment-only and text-plus-attachment messages are idempotent by `clientMessageId` plus attachment fingerprint, and shared-file/media side effects are refreshed from the canonical message only.
   - Acceptance: Backend and frontend tests prove duplicate attachment sends produce one persisted message, one asset entry per file, one sender bubble, and a conflict when the same `clientMessageId` is reused with different attachment content.

8. **Realtime recipient receive**: An authenticated recipient with an active socket receives new messages through Socket.IO without refreshing the page.
   - Current: Socket rooms and `message:new` listeners exist, but production shows recipients may need refresh.
   - Target: If both users are authenticated and connected, sender HTTP create emits a canonical `message:new` that appears in the recipient message list and conversation list without manual refresh.
   - Acceptance: A two-client socket or browser smoke test sends one message from user A to user B and verifies user B receives and renders it without navigation or refresh, while a non-member socket receives no message.

9. **Reconnect reconciliation**: Socket reconnect and `socket:ready` repair stale state without introducing duplicates.
   - Current: The socket hook invalidates chats, unread counts, and active messages on readiness/reconnect, but this is not proven against the production failure.
   - Target: After reconnect, selected chat messages, conversation list latest message, and unread counts refetch and merge into the same canonical identities.
   - Acceptance: A test simulates disconnect, message creation while disconnected, reconnect, and refetch; the recipient sees the missing message once and existing messages are not duplicated.

10. **Server-truth delivery state**: HTTP success or optimistic render never means delivered.
   - Current: Production shows sender UI can imply delivery while the recipient has not received the message.
   - Target: New messages start as `sent`; `delivered` is shown only after recipient-side socket delivery/join/acknowledgement updates server state; `read` is shown only after recipient-side read behavior updates server state.
   - Acceptance: Tests prove sender HTTP success leaves status at `sent` when the recipient is offline, recipient socket delivery promotes to `delivered`, recipient read promotes to `read`, and the sender cannot mark their own message delivered or read.

11. **Conversation list and unread consistency**: Canonical message events update chat lists and unread counts without duplicate rows or false sender unread state.
   - Current: Chat list latest-message and unread updates exist, but the production bug indicates the event stream is not reliable enough.
   - Target: Sender and recipient conversation lists update from the same canonical message identity; recipient unread changes when appropriate, sender unread does not drift, and no duplicate chat rows are introduced.
   - Acceptance: Tests verify a new message updates latest message and ordering for both users, increments recipient unread when the chat is not actively read, does not increment sender unread, and leaves one row per chat.

12. **Production commit alignment**: Completion evidence must prove the tested frontend and backend are the deployed versions being evaluated.
   - Current: Phase 10 production smoke was blocked without complete credentials and production defects were observed in the live app.
   - Target: Verification records the Vercel frontend URL/commit and Render backend URL/commit or deploy identifier before claiming Phase 10.1 complete.
   - Acceptance: Verification artifacts include deployed frontend/backend identifiers and the exact smoke result; if credentials or deployment access are unavailable, the phase is marked blocked instead of complete.

13. **Redacted delivery observability**: Message lifecycle diagnostics exist without exposing private content or secrets.
   - Current: Backend logging exists, but delivery-specific production diagnosis lacks a locked redacted trace contract.
   - Target: The message send lifecycle can be traced by request id or event id, chat id, message id, `clientMessageId`, user role, and lifecycle stage, without logging message text, attachment contents, cookies, tokens, emails, or reset codes.
   - Acceptance: Code review verifies delivery logs contain only redacted metadata, and a smoke run can correlate create, persist, emit, recipient receive/status, and reconciliation stages without private message content.

## Boundaries

**In scope:**
- Backend message creation idempotency for text and existing attachment sends.
- Frontend optimistic, HTTP, socket, retry, refetch, pagination, and reconnect message merge behavior.
- Composer duplicate-submit prevention for a single draft.
- Realtime `message:new` delivery to authenticated chat members.
- Server-truth `sent`, `delivered`, and `read` state requirements.
- Conversation list latest-message and unread-count consistency tied to message delivery.
- Security checks that prevent non-members from receiving messages or updating receipts.
- Local automated tests and two-account smoke coverage for the delivery contract.
- Deployed smoke evidence or an explicit blocked status if production credentials/deploy access are unavailable.
- Redacted delivery observability needed to diagnose this reliability class.

**Out of scope:**
- Call and video-call implementation - separate feature surface, not required to repair message delivery.
- Block/unblock user actions - separate conversation control phase.
- Profile image changing and user avatar management - separate account/profile feature.
- Side rail close/search/more UX fixes - real functional gaps, but not part of the urgent delivery reliability contract.
- New voice-message recording implementation - separate media/composer feature.
- New media gallery or shared-file UX features beyond refreshing existing side effects from canonical messages.
- Automatic destructive cleanup of historical duplicate production messages - data repair must be planned separately after future duplicates are prevented.
- Replacing React/Vite, Express, MongoDB/Mongoose, Socket.IO, TanStack Query, Zustand, or Tailwind - keep the existing stack.
- Re-enabling socket-based message persistence - one create path is required for this phase.

## Constraints

- Preserve the existing MERN, Socket.IO, TanStack Query, Zustand, Tailwind, and npm package layout.
- Preserve cookie-authenticated HTTP and socket identity boundaries; all message and receipt updates must verify chat membership.
- Do not overwrite unrelated local work, especially existing local work in `Frontend/Chatify/src/pages/chat/chat.tsx`, except for the focused changes required by this phase.
- Do not log private message text, attachment contents, cookies, tokens, emails, OAuth payloads, reset codes, or raw cookie metadata.
- Keep production CORS, cookie, and socket credential behavior aligned between Render backend and Vercel frontend.
- Existing users and existing stored messages must remain readable; this phase prevents future duplicate sends but does not perform destructive historical repair.

## Acceptance Criteria

- [ ] Repeated and concurrent sends with the same `clientMessageId` and same payload leave exactly one `Message` document.
- [ ] A normal browser send without `clientMessageId` is rejected rather than persisted.
- [ ] Socket `message:send` does not persist a message; HTTP create remains the only create path.
- [ ] Optimistic, HTTP success, sender socket echo, refetch, pagination, and retry produce one rendered sender bubble.
- [ ] Rapid duplicate composer submissions for the same draft trigger one create mutation and one optimistic message.
- [ ] Failed retry with the same unchanged content resolves into the same message identity and does not append another bubble.
- [ ] Attachment-only and text-plus-attachment duplicate sends produce one persisted message and one asset representation per file.
- [ ] Recipient receives a new message through Socket.IO without refresh in a two-account test.
- [ ] Non-member sockets do not receive `message:new` and cannot mark messages delivered/read.
- [ ] Reconnect/refetch reconciles missed messages without duplicates.
- [ ] Sender HTTP success shows `sent`, not `delivered`, until recipient-confirmed server state promotes it.
- [ ] Recipient read behavior promotes messages to `read`; sender cannot mark their own message delivered or read.
- [ ] Conversation list latest message, order, and unread counts update correctly for sender and recipient without duplicate chat rows.
- [ ] Verification records local automated results, local two-account smoke results, and deployed smoke results with frontend/backend deploy identifiers, or explicitly marks deployed smoke blocked.
- [ ] Delivery lifecycle logs are redacted and contain enough metadata to correlate create, persist, emit, receive/status, and reconciliation stages.
