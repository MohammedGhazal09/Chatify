## Local Two-Account Delivery Smoke

Spec:

```text
Frontend/Chatify/e2e/chat-delivery-reliability.spec.ts
```

Command:

```powershell
cd Frontend\Chatify
$env:CHATIFY_LOCAL_DELIVERY_SMOKE='1'
$env:CHATIFY_LOCAL_BACKEND_URL='http://127.0.0.1:5000'
$env:VITE_BACKEND_URL='http://127.0.0.1:5000'
$env:VITE_SOCKET_URL='http://127.0.0.1:5000'
npm run test:ui -- --grep "Phase 10.1 local delivery reliability"
```

Local backend setup:

- The default configured localhost port `3000` was occupied by a different local app, so the smoke used an isolated Chatify backend on `127.0.0.1:5000`.
- A first isolated backend attempt against the existing local MongoDB configuration was blocked by database connectivity: `users.findOne()` buffering timeout.
- A temporary Chatify backend was then started against `mongodb-memory-server` with `FRONTEND_ORIGIN_DEV=http://127.0.0.1:4177`.
- The temporary backend health check returned HTTP 204 from `/api/csrf-token`.
- The temporary backend process was stopped after the smoke.

Outcome: passed.

Evidence:

- One generated sender account and one generated recipient account were provisioned in the temporary in-memory database.
- The sender opened/continued a direct chat with the recipient through the real UI.
- The recipient opened/continued the same direct chat through a second browser context.
- One unique redacted Phase 10.1 marker was sent by the sender through the real composer and HTTP create path.
- The HTTP create response returned `status: sent`, proving the request success itself did not claim recipient delivery.
- The sender conversation rendered exactly one marker row.
- The recipient conversation received the marker without navigation or refresh.
- Sender and recipient refresh parity checks both rendered exactly one marker row.
- The spec did not install Chatify `page.route` mocks.

Final local smoke status: passed.

## Delivery Diagnostics

## Residual Risks

- Production delivery reliability remains unproven until the opt-in live smoke runs with two real production test accounts.
- The local smoke proves the repaired flow against real browser, HTTP, Socket.IO, and MongoDB paths, but it used a temporary in-memory MongoDB instead of the deployed database.
- The first backend full-test run exposed a transient `mongodb-memory-server` startup timeout; the immediate rerun passed, but the suite remains sensitive to memory-server startup timing on this machine.

## Final Status

Phase 10.1 code-level and local browser reliability gates are passed.

Production evidence is blocked, not passed.

Do not claim deployed Chatify delivery reliability complete until `CHATIFY_PRODUCTION_SMOKE=1` production smoke passes and deployed identifiers are recorded.
