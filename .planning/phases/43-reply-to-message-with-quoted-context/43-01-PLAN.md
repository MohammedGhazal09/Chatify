# Phase 43 Plan 01 - Backend Reply Metadata Contract

## Objective

Add the durable backend reply metadata contract for standard messages while preserving message visibility, idempotency, pagination, socket payloads, and encrypted-mode privacy.

## Scope

- Add a `replyTo` subdocument to the message model with bounded quote metadata.
- Add reply source validation to `newMessage` for standard conversations.
- Add serializer support for `replyTo` in `messageState.mjs`.
- Add reply fingerprinting to standard message idempotency.
- Keep encrypted replies disabled or metadata-only without plaintext leakage.
- Add focused backend tests for success, visibility failures, deleted sources, encrypted limitation, and idempotency conflicts.

## Acceptance Criteria

- [x] `POST /api/message/new-message` accepts `replyToMessageId` for a visible same-chat standard source message.
- [x] Response, history, search context, and socket `message:new` payloads include stable `replyTo`.
- [x] Unauthorized, hidden, deleted-for-everyone, malformed, and out-of-chat reply sources return generic failure without leaks.
- [x] Same `clientMessageId` plus same reply target is idempotent.
- [x] Same `clientMessageId` plus different reply target returns 409.
- [x] Encrypted conversations do not accept server-readable quote plaintext.
