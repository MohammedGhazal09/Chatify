# Phase 10.1 Review Fix

## Fixed Findings

### CR-10.1-01 - Idempotent retries bypass delivery side-effect repair

Status: Fixed.

Changes:
- Added a shared message-create finalizer in `Backend/Chatify/Controller/messageController.mjs`.
- Routed fresh creates, pre-found idempotent retries, and duplicate-key retry convergence through the same finalizer.
- The finalizer safely re-runs latest-message repair, canonical `message:new` emission, and recipient unread-count emission.
- Latest-message repair now compares the current chat latest message before updating, so an old idempotent retry cannot move the chat list backward after a newer message exists.
- Added HTTP idempotency regression coverage for repairing a stale `Chats.latestMessage`.
- Added HTTP idempotency regression coverage for preserving a newer `Chats.latestMessage` during an older retry.
- Updated the socket contract test to expect a retry to re-emit canonical side effects while persistence remains single-copy.

Rationale:
- A retry with the same `clientMessageId` is now able to repair the exact stale side-effect state that causes refresh-only recipient delivery.
- Re-emitting the canonical message is safe because the frontend cache deduplicates by `_id` and `clientMessageId`.

### WR-10.1-01 - Production smoke credentials can leak into Playwright failure artifacts

Status: Fixed.

Changes:
- Added `authenticateProductionSmokeContext` to `Frontend/Chatify/e2e/pages/productionSmoke.ts`.
- Production smoke specs now login through backend API `fetch`, parse the auth cookie, and inject it into the Playwright browser context before opening the UI.
- Removed browser-form credential filling from:
  - `Frontend/Chatify/e2e/chat-production-delivery-reliability.spec.ts`
  - `Frontend/Chatify/e2e/chat-production-reality.spec.ts`
- Added context cleanup for authentication/page boot failures.

Rationale:
- Real smoke passwords are no longer typed into browser fields that can appear in Playwright trace snapshots.
- The smoke audit still uses redacted account metadata.

### WR-10.1-02 - Local smoke can create persistent generated users on nonlocal backends

Status: Fixed.

Changes:
- Removed fallback from `CHATIFY_LOCAL_BACKEND_URL` to frontend `.env` `VITE_BACKEND_URL`.
- Required an explicit `CHATIFY_LOCAL_BACKEND_URL`.
- Rejected non-loopback backend URLs by default.
- Added `CHATIFY_ALLOW_NONLOCAL_DELIVERY_SMOKE=1` as an explicit nonlocal opt-in.
- Required explicit smoke accounts for nonlocal runs.
- Required `CHATIFY_LOCAL_EPHEMERAL_BACKEND=1` before generated local accounts can be used.

Rationale:
- A local smoke run can no longer silently use a shared/staging/production-like backend from frontend env.
- Generated accounts are constrained to deliberately marked ephemeral backends unless the caller supplies durable smoke accounts.
