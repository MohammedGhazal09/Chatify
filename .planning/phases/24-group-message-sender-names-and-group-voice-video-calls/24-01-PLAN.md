# Plan 24-01: Group Sender Labels And Frontend Availability

## Objective

Make group messages visibly attributable and make frontend call availability group-aware without creating dead controls.

## Must-Haves

- Cover `24-SPEC.md` requirements 1 and 2.
- Cover `24-CONTEXT.md` decisions D-01 through D-06 and D-14 through D-19.
- Preserve direct one-to-one call availability behavior.
- Do not add socket listeners or call transport logic inside presentational components.

## Tasks

1. Keep or add sender display-name resolution in `MessageBubble.tsx`.
2. Render the sender name above every normal group message bubble with truncation and existing muted metadata styling.
3. Add component coverage for group sender labels.
4. Add group availability logic in `useCallController.ts` that allows an eligible group chat when at least one other member is reachable through existing presence data.
5. Preserve concrete unavailable reasons for disconnected socket, unsupported WebRTC, active call, disabled conversation activity, and no reachable group member.
6. Add frontend tests for enabled group audio/video availability and unavailable group states.

## Verification

- `cd Frontend/Chatify; npm test -- MessageBubble.test.tsx useCallController.test.tsx`
- `cd Frontend/Chatify; npm run lint`
- `cd Frontend/Chatify; npm run build`

## Success Criteria

- Group message sender labels are visible in tests.
- Group call controls can become available for eligible group chats.
- Ineligible group calls expose concrete reasons instead of a direct-chat-only blanket message.
- Existing direct call controller tests still pass.
