# Plan 18-04: Regression Guards And Operational Evidence

## Objective

Lock the new observability and runbook surfaces behind automated checks and record a final operational readiness artifact without overstating production release status.

## Tasks

1. Read `18-VALIDATION.md`, the new logger/readiness tests, root scripts, runbooks, `.env.example` files, and Phase 17 readiness artifact.
2. Add `scripts/ops-check.mjs` that validates required env example keys, runbook presence, root script presence, and forbidden secret-shaped content in Phase 18 operational docs and tests.
3. Wire `npm run ops:check` to the guard script.
4. Run targeted backend observability tests, full backend tests, frontend tests, lint, build, root quality, and ops guard where feasible.
5. Create `18-OPERATIONS-READINESS.md` with command results, endpoint coverage, docs coverage, privacy scan results, and remaining release blockers inherited from Phase 14, Phase 15, and Phase 17.
6. Create `18-01-SUMMARY.md` through `18-04-SUMMARY.md` as plans complete, then update roadmap/state.

## Verification

- `npm run ops:check` fails if required env examples, runbooks, root scripts, or privacy guardrails drift.
- `18-OPERATIONS-READINESS.md` exists and records exact command outcomes.
- The artifact explicitly states that Phase 18 does not unblock v1 release without production smoke and call/TURN evidence.
- `gsd-tools query verify phase-completeness 18` reports 4 plans and 4 summaries after execution.

## Success Criteria

Phase 18 ends with durable operational evidence and automated guards, while production readiness remains blocked unless the missing live evidence is supplied.

## Must Haves

- Verification evidence must be command-backed.
- No final artifact may claim Chatify is launch-ready from local operations hardening alone.
