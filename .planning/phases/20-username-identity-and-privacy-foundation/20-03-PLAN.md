# Plan 20-03: Auth Propagation, Privacy Guardrails, And Username Verification Evidence

## Objective

Finish Phase 20 by propagating username through account-safe auth state, removing email from public identity surfaces, and recording executable verification evidence for username readiness before Phase 21.

## Tasks

1. Read `20-SPEC.md`, `20-CONTEXT.md`, `20-UI-SPEC.md`, `20-01-PLAN.md`, `20-02-PLAN.md`, identity serializers, chat/user/socket payload tests, frontend auth/chat types, and fixture leak guards.
2. Create or tighten account-safe vs public-safe user serializers so logged-in owner responses may include email, while public identity/contact/presence/socket payloads do not.
3. Update identity event serialization, public user/contact responses, socket identity payloads, and frontend types/fixtures to include username where needed and exclude email outside owner account/auth/reset contexts.
4. Add backend response tests proving public identity, online/contact/status, chat participant, and identity update payloads do not contain email and do include username when available.
5. Add or update frontend fixture/privacy tests proving discovery-facing and identity-facing fixtures do not rely on email as a public display value.
6. Run focused backend and frontend test commands from plans 20-01 and 20-02, then run lint/build gates or document blockers honestly.
7. Write `20-VERIFICATION.md` with exact commands, outcomes, changed privacy boundaries, and any remaining blockers. Include a focused `rg` evidence section for new email exposure paths.
