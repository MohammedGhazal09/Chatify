# Phase 10.1 Code Review

## Scope

Reviewed the Phase 10.1 backend message/socket delivery contract, frontend message cache/socket/send guard convergence, and the new local/production Playwright delivery reliability smoke coverage.

## Findings

### CR-10.1-01 - Idempotent retries bypass delivery side-effect repair

- **Severity:** Critical
- **Files:** `Backend/Chatify/Controller/messageController.mjs:480`, `Backend/Chatify/Controller/messageController.mjs:504`, `Backend/Chatify/Controller/messageController.mjs:555`, `Backend/Chatify/Controller/messageController.mjs:569`
- **Problem:** `newMessage` returns an existing canonical message from both idempotent paths before running the post-create side effects. The normal path updates `Chats.latestMessage` and emits `message:new`/unread updates after persistence, but an idempotent retry exits at the early `200` response instead. If the first request creates the message and then the process fails, throws, or loses the socket emit before completing `latestMessage`/unread/realtime side effects, a retry with the same `clientMessageId` will confirm success without repairing those side effects. That leaves exactly the class of stale chat list / recipient refresh-only delivery state Phase 10.1 is meant to eliminate.
- **Recommendation:** Route fresh creates and idempotent duplicate responses through a shared side-effect finalizer that is safe to re-run: set `latestMessage`, recompute unread count events, and emit the canonical `message:new` payload. The frontend cache already dedupes by `clientMessageId`/`_id`, so a repeated canonical socket event is safer than a missed one. Add a regression where a message already exists with matching `clientMessageId` but `Chats.latestMessage` is stale, then assert the retry repairs latest-message state and emits the canonical event.

### WR-10.1-01 - Production smoke credentials can leak into Playwright failure artifacts

- **Severity:** Warning
- **Files:** `Frontend/Chatify/e2e/chat-production-delivery-reliability.spec.ts:19`, `Frontend/Chatify/e2e/chat-production-delivery-reliability.spec.ts:20`, `Frontend/Chatify/playwright.config.ts:11`
- **Problem:** The production smoke logs in by filling the real smoke account email and password into browser form fields while the global Playwright config enables `trace: 'on-first-retry'`. On a failing retry, trace snapshots and error context can capture page state and typed values. The report body redacts emails, but the raw Playwright artifacts are a separate leak path for production smoke account credentials and PII.
- **Recommendation:** For production smoke, avoid typing secrets into traced pages. Prefer API login that stores authenticated browser state/cookies before opening the UI, or disable trace/video/screenshot for the production smoke project and explicitly scrub any failure artifacts before upload. Keep the markdown audit redaction, but treat Playwright traces as secret-bearing unless proven otherwise.

### WR-10.1-02 - Local smoke can create persistent generated users on nonlocal backends

- **Severity:** Warning
- **Files:** `Frontend/Chatify/e2e/chat-delivery-reliability.spec.ts:82`, `Frontend/Chatify/e2e/chat-delivery-reliability.spec.ts:94`, `Frontend/Chatify/e2e/chat-delivery-reliability.spec.ts:127`
- **Problem:** The local smoke falls back from `CHATIFY_LOCAL_BACKEND_URL` to `VITE_BACKEND_URL`, then auto-generates and signs up accounts when explicit local smoke users are not provided. If `VITE_BACKEND_URL` points to a shared, staging, or production-like backend, running `CHATIFY_LOCAL_DELIVERY_SMOKE=1` can create permanent test users, chats, and messages outside the intended isolated environment. The test only closes browser contexts; it does not clean up backend data.
- **Recommendation:** Require `CHATIFY_LOCAL_BACKEND_URL` for the local smoke and reject non-loopback origins by default. If nonlocal execution is ever needed, gate it behind a deliberately named opt-in such as `CHATIFY_ALLOW_NONLOCAL_DELIVERY_SMOKE=1` and require explicit smoke accounts. For generated accounts, either run only against an ephemeral memory-backed backend or add a cleanup path that deletes generated users/chats/messages after the test.
