# Plan 19-01: Notification Preference And Privacy Model

## Objective

Create the tested notification domain that later UI and socket wiring can depend on: typed preferences, per-user local persistence, permission-state helpers, mute helpers, and privacy-safe notification copy.

## Tasks

1. Read `19-CONTEXT.md`, `19-RESEARCH.md`, `sounds.ts`, `useSelectedChatPersistence.ts`, `SettingsModal.tsx`, and `useChatSocket.ts`.
2. Add notification types for sound enabled, browser notification opt-in, browser permission state, unsupported state, and muted chat ids.
3. Add a privacy helper that formats browser/in-app notification copy without raw message text, attachment names, emails, tokens, or reset-code-shaped strings.
4. Add a per-user notification preference hook or utility that namespaces localStorage by authenticated user id, migrates or preserves the existing sound preference intentionally, and exposes mute/unmute helpers.
5. Add browser `Notification` permission helpers that handle unsupported browsers, `default`, `denied`, and `granted` states without requesting permission on render.
6. Add focused tests for per-user preference isolation, muted chat behavior, permission state handling, safe-copy formatting, and localStorage failure tolerance.

## Verification

- `cd Frontend/Chatify; npm test -- --run src/utils/notificationPrivacy.test.ts src/hooks/useNotificationPreferences.test.tsx`
- A grep of notification helper tests and Phase 19 evidence fixtures contains no realistic email, token, reset code, or private message text in expected browser notification output.

## Success Criteria

The project has a reusable notification preference and privacy layer with tests before any UI or socket behavior starts depending on it.

## Must Haves

- Browser notification title/body must never include `message.text`, attachment display names, raw email addresses, or token-like values.
- Preferences must not bleed across local accounts on the same browser profile.
- Muting affects alerts only, not message storage, unread counts, or realtime receipts.
