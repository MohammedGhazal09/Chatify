# Phase 38 Plan 01 - Backend Space And Channel Data Contract

## Objective

Add private space/channel persistence, role-aware membership controls, and server-owned serialization without email exposure.

## Scope

- `Space` model with name, description, members, roles, createdBy, and limits.
- Channel metadata model or chat-backed channel metadata, depending on the lowest-risk fit with existing `Chat`.
- Space routes/controllers for create, list, detail, add member, remove member, create channel, and list channels.
- Role helpers for owner/admin/member authorization.
- Public identity serialization for space members without email.
- Tests for role checks, private listing, member removal, channel access, and validation.

## Acceptance Criteria

- [ ] Authenticated user can create a private space with a default channel.
- [ ] Non-members cannot list or fetch a space/channel.
- [ ] Owner/admin can add members by username and remove non-owner members.
- [ ] Email is never accepted as the member identifier or returned in space/member responses.
- [ ] Space/channel limits and field validation reject unsafe input.
